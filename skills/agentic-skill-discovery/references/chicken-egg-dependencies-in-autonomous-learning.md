# The Chicken-Egg Problem in Autonomous Learning: When Evaluation Depends on What You're Trying to Learn

## The Circular Dependency at the Heart of Open-Skill Discovery

Traditional reinforcement learning assumes a well-defined reward function and success criteria provided by humans. You know what "opening a drawer" means, you can write code to check if the drawer is open, and you can shape rewards to encourage drawer-opening behaviors.

Autonomous skill discovery breaks this assumption. The system must:
1. Decide what skill to learn (task proposal)
2. Define what success means (success function)
3. Determine how to incentivize learning (reward function)
4. Evaluate whether learning succeeded (fitness measurement)

Each step depends on the others, creating circular dependencies that can trap the system in false learning cycles or prevent learning entirely.

The ASD paper identifies this explicitly: "In open-skill learning, where the focus is on optimizing newly proposed tasks rather than predefined ones, naive Eureka-like methods [14] cannot be directly applied due to the fact that the success criteria cannot be predetermined... As a result, evolution selection becomes impossible due to the absence of a unified golden metric to quantify performance."

And more starkly: "The success function and reward function form a chicken-egg relation in that (1) the reward function search relies on reliable success determination and, meanwhile, (2) it is unfeasible to verify the success function before the learning."

## Anatomy of the Circular Dependencies

### Loop 1: Reward Search Requires Success Determination

Evolutionary reward function search works by:
1. Sample candidate reward functions
2. Train RL policies using each candidate
3. Measure success rate of resulting policies
4. Keep best-performing reward functions
5. Mutate and repeat

Step 3 requires evaluating "success rate"—what fraction of rollouts achieve the task? This requires a success function that can reliably detect task completion.

**The problem**: For open-skill learning, the success function is *also* being generated by the LLM. There's no ground truth to verify it against. The LLM might generate:

```python
def success_open_drawer(obs):
    return obs['drawer_position'][0] > 0.05  # Drawer moved 5cm
```

But what if the drawer can actually move 10cm? This success function will fire prematurely, labeling partially-successful behaviors as complete successes, causing the evolutionary search to optimize for the wrong behaviors.

Or worse:

```python
def success_open_drawer(obs):
    return obs['drawer_position'][0] > 0.15  # Drawer moved 15cm (but max is 10cm)
```

Now the success function never fires (impossible criterion), all behaviors are labeled failures, good reward functions are discarded, and learning stalls.

**The dependency**: Can't search for reward functions without reliable success determination. Can't verify success determination without observing learned behaviors. Can't observe learned behaviors without reward functions.

### Loop 2: Evolution Requires Fitness Measurement

Evolutionary search relies on fitness functions to guide selection. The ASD framework uses success rate as fitness:

```python
fitness = fraction_of_rollouts_that_succeed
```

This seems straightforward until you realize:
- Success is determined by LLM-generated success functions (potentially wrong)
- The same LLM generating reward functions (could have correlated errors)
- There's no external ground truth to calibrate against

If the success function is too lenient, fitness scores are inflated, bad reward functions survive, and the skill library gets contaminated with false positives.

If the success function is too strict, fitness scores are deflated, good reward functions are eliminated, and the library fails to grow despite capable behaviors existing.

**The dependency**: Can't evolve reward functions without fitness. Can't compute fitness without success determination. Can't trust success determination from the same system generating reward functions.

### Loop 3: Task Proposal Depends on Current Capabilities

The LLM proposes new tasks based on:
- Environment description
- Robot capabilities
- Previously learned skills
- Previously failed attempts

From the paper: "The LLM will be informed about tasks that could not be completed so that it will have a sense of the limits of the learning agent, influencing the successive task proposals."

This creates a feedback loop:
- Contaminated skill library → LLM thinks robot is more capable than it is → proposes impossible tasks → more failures
- False negatives in verification → LLM thinks robot is less capable → proposes redundant simple tasks → learning stalls

**The dependency**: Task proposal quality depends on accurate capability assessment. Capability assessment depends on reliable success verification. Success verification depends on trustworthy success functions. Success functions come from the same LLM proposing tasks.

## How ASD Breaks the Cycles: Architectural Interventions

The paper doesn't solve these dependencies by eliminating them—that's impossible. Instead, it introduces architectural separations and redundancy to prevent cascading failures:

### Intervention 1: Sample Multiple Success Functions

Rather than generating one success function and trusting it:

"With a temperature of 1.0, the LLM samples three success functions in each iteration; based on each, three reward functions are further sampled."

This creates 3 × 3 = 9 parallel learning tracks. Even if some success functions are wrong, others might be correct. The system hedges against single-function failure.

**Why this helps**: Breaks the dependency by introducing redundancy. Not all sampled success functions will have the same errors. If one is too lenient and one is too strict, the third might be approximately correct.

**Limitation**: If the LLM has systematic bias (e.g., always underestimates drawer travel distance), all sampled functions might be wrong in the same direction.

### Intervention 2: Independent Slow Verification (VLM)

The key architectural innovation: use a *different model* (VLM, not LLM) to verify success after learning:

"We prompt an independent VLM to additionally examine the success of survivor candidates, before passing them on to the next evolutionary generation."

The VLM:
- Uses different modality (vision vs. state coordinates)
- Has different training data (vision-language pairs vs. code)
- Runs at different time scale (post-training vs. during training)
- Makes different errors (won't have correlated mistakes with LLM)

**Why this helps**: Breaks the circular dependency by introducing external validation. The VLM doesn't depend on the LLM's success function—it observes actual behavior and makes an independent judgment.

**How it breaks the cycle**:
```
Without VLM:
LLM writes success function → RL trains → LLM-success judges → if pass, accept skill
[Same model judges its own success criteria → circular]

With VLM:
LLM writes success function → RL trains → LLM-success judges → VLM judges → if both pass, accept skill
[Independent verification breaks circle]
```

### Intervention 3: Asymmetric Verification Investment

The paper applies slow verification only to P₁ candidates (those passing fast verification), not to all behaviors:

"Since the amount of unsuccessful behaviors during learning (system-1 negatives) is overwhelming to evaluate, and N₁P₂ samples are less common and less harmful, we do not additionally examine them due to the high assessment cost."

This asymmetry is critical for tractability. If you had to run VLM verification on *every* rollout during *every* RL training step, the compute cost would be prohibitive.

**Why this helps**: Focuses expensive verification where it matters most (preventing false positives from contaminating library) while accepting false negatives (which only slow learning, not corrupt it).

**The trade-off**: Some correct behaviors might be rejected (false negatives), but this is better than accepting incorrect behaviors (false positives) that pollute the skill library.

### Intervention 4: RAG as Accumulated Ground Truth

After skills are verified by both systems, they're added to the retrieval database:

"We apply the RAG (Retrieval Augmented Generation) technique to retrieve validated function references for the efficacy of LLM reasoning."

This accumulated library of double-verified success and reward functions serves as progressively refined ground truth:

**Why this helps**: Early in learning, the system has no ground truth—it's bootstrapping from scratch. But each verified skill adds a data point. Over time, the retrieval database contains enough examples that new function generations can be validated against *patterns* rather than trusting the LLM blindly.

**How it breaks cycles**: Instead of circular dependency (new function depends on itself being correct), you get accumulating validation (new function depends on patterns from previous validated functions).

**Example trajectory**:
- Iteration 1: Learn "reach cube," verify with VLM → add to database
- Iteration 2: Learn "grasp cube," retrieve "reach cube" pattern → LLM sees working example → generates similar success function → verify with VLM → add to database
- Iteration 3: Learn "lift cube," retrieve "grasp cube" pattern → LLM sees grasping detection → generates similar function → verify → add
- Iteration N: Library contains rich patterns for manipulation tasks → new tasks bootstrap from proven patterns

## The False Learning Cycles: What Happens When Dependencies Aren't Broken

Figure 7 in the paper illustrates the failure modes:

### False Positive Cycle (P₁N₂ contamination)

1. **LLM generates lenient success function**:
   ```python
   def success_open_drawer(obs):
       return obs['drawer_pos'][0] > 0.02  # Only 2cm, but drawer can open 10cm
   ```

2. **RL learns policy that moves drawer 3cm**:
   - This passes the lenient success function
   - It's a local optimum: easy to achieve, satisfies success criterion
   - But it doesn't actually accomplish the task (drawer isn't fully open)

3. **Evolutionary search selects this reward function**:
   - Success rate is high (passes lenient check)
   - Reward function gets high fitness score
   - Becomes template for next generation

4. **Skill added to library with false confidence**:
   - Library now contains "open drawer" skill that only opens drawer 3cm
   - Future tasks depending on "open drawer" fail mysteriously
   - LLM proposes tasks assuming drawer can be opened

5. **Cascading failures**:
   - "Place cube in drawer" task proposed
   - Decomposes to: "open drawer" → "grasp cube" → "place in drawer"
   - First step uses contaminated skill (opens only 3cm)
   - Third step fails (drawer not open enough)
   - LLM can't diagnose why (thinks drawer was opened)

6. **Library corruption spreads**:
   - LLM sees "place in drawer" as achievable (uses "open drawer" skill)
   - Proposes more drawer-related tasks
   - All fail due to contaminated "open drawer" skill
   - But failures are attributed to other causes

**The death spiral**: False positives accumulate, library increasingly contains skills that seem to work but don't, future tasks increasingly fail for mysterious reasons, system loses ability to learn new skills because it can't build on contaminated foundation.

### False Negative Cycle (Correct Skills Rejected)

1. **LLM generates strict success function**:
   ```python
   def success_grasp_cube(obs):
       return (obs['cube_pos'][2] > 0.1 and  # Cube lifted 10cm
               obs['gripper_closed'] > 0.9 and  # Gripper fully closed
               obs['gripper_force'] > 5.0)  # Strong grip force
   ```

2. **RL learns policy that successfully grasps**:
   - Lifts cube 8cm (good enough for practical grasping)
   - Gripper closed 0.85 (secure grip)
   - Force 4.5N (adequate)

3. **Success function rejects this behavior**:
   - Doesn't meet all criteria (force too low)
   - Success rate is zero despite functional grasping
   - Associated reward function gets fitness = 0

4. **Evolutionary search eliminates good reward functions**:
   - All reward functions that produce functional grasping get low fitness
   - Evolution selects reward functions that produce unsuccessful behaviors (that accidentally meet arbitrary criteria)

5. **Learning stalls**:
   - No successful policies found despite extensive search
   - Task added to failure pool
   - LLM concludes "grasping is too hard with current capabilities"

6. **Capability estimation corrupted**:
   - LLM stops proposing tasks requiring grasping
   - Simpler tasks proposed (just touching, reaching)
   - System underestimates its own capabilities
   - Learning plateaus below potential

**The stagnation spiral**: Overly strict verification prevents learning, capability set appears smaller than reality, task proposals become overly conservative, system stops exploring, potential capabilities never discovered.

## Design Patterns for Breaking Circular Dependencies

### Pattern 1: Independence Through Diversity

When A depends on B and B depends on A, introduce C that:
- Uses different information than A or B
- Makes different errors than A or B
- Operates at different time scales than A or B

In ASD: LLM (text-based, code generation) → VLM (vision-based, behavior observation)

For agent orchestration:
- Skill selection (LLM reasoning) → Execution monitoring (rule-based anomaly detection)
- Task decomposition (semantic understanding) → Verification (formal methods checking preconditions)
- Performance estimation (learned model) → Validation (statistical testing)

### Pattern 2: Redundancy with Diversity

When you can't trust a single evaluation, sample multiple diverse evaluations:

In ASD: Sample 3 success functions × 3 reward functions = 9 learning tracks

For agent orchestration:
- Multiple decomposition strategies (different LLMs, different prompt templates)
- Multiple execution plans (different skill chains)
- Multiple validation methods (code review, testing, formal verification)

**Critical**: Diversity matters more than quantity. Sampling 9 success functions from same LLM with same prompt isn't much better than sampling 1. But sampling from different LLMs, or with different temperature settings, or with different prompt phrasings creates genuine diversity.

### Pattern 3: Asymmetric Investment in Verification

Expensive verification where failures are costly, cheap verification where failures are tolerable:

In ASD:
- Fast verification (cheap, every RL step): prevent obviously bad behaviors
- Slow verification (expensive, only survivors): prevent library contamination

For agent orchestration:
- Real-time monitoring (cheap, every skill execution): catch crashes, timeouts
- Post-execution validation (expensive, only for library addition): ensure correctness
- Periodic auditing (very expensive, sample of deployed skills): detect degradation

### Pattern 4: Progressive Ground Truth Accumulation

Bootstrap with no ground truth, accumulate verified examples, use examples as ground truth:

In ASD:
- Initially: no verified skills, must trust LLM
- After N skills: retrieval database contains patterns
- Eventually: new functions generated from proven patterns

For agent orchestration:
- Initially: rely on human-provided examples
- After deployment: accumulate successful execution traces
- Eventually: use traces to validate new compositions

**The key transition**: At some point, the accumulated verified examples become more trustworthy than the LLM generating new candidates. The system shifts from trusting generation to trusting retrieval-based generation.

### Pattern 5: Explicit Doubt Representation

Rather than treating evaluations as binary (success/failure), represent uncertainty:

```python
class SkillEvaluation:
    def __init__(self):
        self.fast_check_passed: bool
        self.slow_check_passed: Optional[bool]  # None if not yet verified
        self.confidence: float  # How certain are we?
        self.evidence: List[str]  # Why do we believe this?
```

This allows:
- Flagging skills with low-confidence verification for human review
- Preferring high-confidence skills when composing
- Identifying when verification is uncertain vs. confident

In ASD, this is implicit in the P₁/P₂/N₂ classification. Making it explicit enables richer decision-making.

## When These Patterns Don't Work: Boundary Conditions

### Systematic Bias in All Evaluators

If LLM and VLM have correlated errors (e.g., both trained on similar data, both fail on same edge cases), independence is illusory.

**Detection**: Track P₁P₂ skills that later fail in execution. If failure rate is high, evaluators have correlated errors.

**Mitigation**: Add third evaluator with different architecture (e.g., rule-based checks, formal verification, human sampling).

### Insufficient Diversity in Sampling

If temperature is too low or prompts too similar, sampled functions lack diversity, redundancy doesn't help.

**Detection**: Measure functional diversity (how different are sampled functions?). If diversity is low despite multiple samples, increase temperature or vary prompts.

**Mitigation**: Explicitly enforce diversity (reject samples too similar to previous samples).

### Expensive Verification Becomes Bottleneck

If slow verification is too expensive, it can't scale to library growth.

**Detection**: Verification time grows faster than linear with library size.

**Mitigation**: 
- Amortize verification cost (batch verification)
- Sample-based verification (don't verify every skill option, verify representative samples)
- Learned verification (train a model to predict VLM judgment, use it for pre-filtering)

### Accumulated Examples Encode Incorrect Patterns

If early false positives contaminate the retrieval database, all future generations inherit the errors.

**Detection**: New skills consistently fail despite being generated from retrieved patterns.

**Mitigation**:
- Periodic database auditing (revalidate stored examples)
- Weight retrieval by actual deployment success (downweight patterns that lead to failures)
- Versioned database (keep track of when examples were added, prefer recent over old)

## The Meta-Lesson: Autonomy Requires Architectural Skepticism

The circular dependencies in autonomous learning reveal a deep principle: **systems that evaluate themselves need architectural mechanisms to prevent self-deception.**

When a system generates solutions and evaluates those solutions using criteria it also generated, you need:

1. **Multiple independent evaluators** (diversity)
2. **Evaluation at multiple time scales** (fast for filtering, slow for commitment)
3. **Asymmetric verification investment** (expensive where it matters)
4. **Progressive ground truth accumulation** (bootstrap then refine)
5. **Explicit uncertainty representation** (know what you don't know)

Without these mechanisms, the system can fall into false confidence (accepting wrong solutions) or false doubt (rejecting correct solutions). Either failure mode prevents learning.

The ASD framework doesn't eliminate circular dependencies—that's impossible in truly autonomous learning. Instead, it breaks the *loops* that would amplify errors into cascading failures. The architecture admits doubt, builds in redundancy, and progressively reduces uncertainty through accumulated evidence.

For WinDAG systems coordinating multiple agents solving complex problems, this suggests: **trust, but verify—and verify differently than you generated.**
