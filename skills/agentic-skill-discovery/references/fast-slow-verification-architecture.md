# Fast and Slow Success Determination: Preventing False Learning Cycles in Autonomous Systems

## The Core Problem: Who Watches the Watchmen?

When an intelligent system must learn skills autonomously—without human-specified reward functions or success criteria—it faces a profound architectural challenge: *how can a system reliably judge its own success when the criteria for success are themselves being generated by that same system?*

The Agentic Skill Discovery (ASD) framework exposes this problem starkly. The system uses an LLM to generate both reward functions (guiding learning) and success functions (evaluating outcomes). As the authors note: "Given that the LLM serves dual roles as both a 'player' (for reward function optimization) and as a 'referee' (for success determination), employing the resultant success rate as the fitness function for evolution may jeopardize learning stability and trustworthiness."

This isn't merely an implementation detail—it reveals a fundamental failure mode in autonomous learning systems: **circular validation**. When the same model that proposes solutions also validates them, systematic biases compound rather than cancel. The paper demonstrates empirically that among all behaviors judged successful by LLM-generated success functions (system-1 positives), only 46.43% are actually correct according to human inspection.

## The Architectural Solution: Dual-Process Verification

Drawing inspiration from Kahneman's dual-process theory of cognition (though not explicitly cited), ASD implements two distinct verification mechanisms operating at different temporal scales and with different computational costs:

### System 1: Fast Success Determination

"Fast success determination (system 1), as opposed to assessing after fixed intervals of execution, as slow success determination (system 2), is pivotal in reinforcement learning."

Fast verification runs at every RL timestep during training. The LLM generates Python success functions that return binary signals: has the task been completed? These functions are:

- **Computationally cheap**: Simple coordinate checks, distance calculations
- **Temporally granular**: Evaluated at every physics step (250 max per episode)
- **Necessary for training**: Enable sparse rewards and early termination
- **Potentially unreliable**: Generated by the same LLM proposing the task

The system samples multiple success function candidates: "With a temperature of 1.0, the LLM samples three success functions in each iteration; based on each, three reward functions are further sampled." This hedges against single-function failures but doesn't eliminate systematic bias.

**Critical insight**: "It is challenging to have an evolutionary search for both [reward and success functions] at the same time" because the reward function search relies on reliable success determination, yet success functions can't be verified until policies are learned—a chicken-egg dependency.

### System 2: Slow Success Determination

After RL training produces candidate policies that pass fast verification, an independent Vision-Language Model (GPT-4V) performs post-training validation:

- **Computationally expensive**: Analyzing video keyframes with large VLM
- **Temporally coarse**: Applied only to survivor candidates from evolution
- **Independent validation**: Different model architecture, different training data
- **High precision**: Achieves 76.50% precision vs. 46.43% for system-1 alone

"We prompt an independent VLM to additionally examine the success of survivor candidates, before passing them on to the next evolutionary generation."

The independence is critical. By using a different model (VLM vs. LLM) trained on different data (vision-language pairs vs. text), system-2 verification breaks the circular dependency. It acts as an external reality check rather than another opinion from the same reasoning process.

## The Four Outcomes: A Confusion Matrix for System Design

The interaction of fast and slow verification creates four possible outcome categories:

| Category | System 1 | System 2 | Interpretation | Frequency |
|----------|----------|----------|----------------|-----------|
| P₁P₂ | Pass | Pass | True positive—reliable skill | Target outcome |
| P₁N₂ | Pass | Fail | False positive—contaminated skill | 23.50% of P₁ |
| N₁P₂ | Fail | Pass | False negative—missed opportunity | Rare |
| N₁N₂ | Fail | Fail | True negative—unsuccessful learning | Common |

The paper only applies slow verification to P₁ candidates because "the amount of unsuccessful behaviors during learning (system-1 negatives) is overwhelming to evaluate, and N₁P₂ samples are less common and less harmful."

This asymmetric verification strategy is pragmatically brilliant: 
- **P₁N₂ is dangerous**: False positives contaminate the skill library, causing future tasks to fail when composed with unreliable skills
- **N₁P₂ is tolerable**: False negatives merely slow learning—the skill wasn't added but could be proposed again
- **N₁N₂ is expected**: Most learning attempts fail; checking all failures is computationally prohibitive

## The Two Failure Modes: Why Independent Verification Matters

Figure 7 in the paper illustrates the cascading failures that occur without slow verification:

### False Positive Cycle (P₁N₂ contamination)
1. LLM generates overly lenient success function
2. RL produces behavior that passes fast check but doesn't actually complete task
3. Skill gets added to library with false confidence
4. Future tasks decompose into subtasks using contaminated skill
5. Complex tasks fail mysteriously because component skills don't work
6. LLM proposes increasingly inappropriate tasks based on false capability assessment

"The LLM-generated success determination may confidently but wrongly assess a task as successful, leading to an undesired false positive learning cycle with a contaminated skill library and non-accomplishable future task proposals."

### False Negative Cycle (correct skills rejected)
1. LLM generates overly strict success function
2. RL produces behavior that actually completes task but fails fast check
3. Low "success rate" causes reward function to be eliminated in evolution
4. No skills learned despite capable behaviors existing
5. Task added to failure pool
6. Skill library fails to expand, limiting future capability

"Negative evaluation of indeed successful behaviors will misguide the reward function search and prevent skill acquisition, resulting in a false negative learning cycle."

The empirical impact is dramatic. Table 1 shows that without slow verification, the system would accept numerous false positives:
- Task 4 (Pick up cube A): 1 false positive out of 5 system-1 positives
- Task 9 (Place plate): 1 false positive, 1 false negative
- Task 13 (Gripper toggle): 1 false positive
- Task 18 (Position cube before drawer): 2 false positives, 0 true positives

## Design Principles for Multi-Agent Orchestration Systems

The fast/slow verification pattern generalizes beyond robotic skill learning to any autonomous system making decisions under uncertainty:

### 1. Separate Evaluation from Execution
Agent systems should not use the same reasoning process to both generate solutions and validate them. When possible:
- Use different model architectures for proposal vs. validation
- Use different knowledge sources (e.g., learned policy vs. VLM observation)
- Apply validation at different abstraction levels (step-wise vs. episode-wise)

### 2. Asymmetric Verification Investment
Not all errors are equally costly:
- Invest expensive verification where false positives create cascading failures (e.g., skills added to permanent library)
- Accept false negatives where they merely slow progress (e.g., rejecting valid one-off solutions)
- Use cheap-but-biased checks for training loops, expensive-but-independent checks for commitment decisions

### 3. Build Verification Budgets into Architecture
The paper's approach works because:
- Fast verification is cheap enough to run in RL inner loop (thousands of times)
- Slow verification is expensive but only applied to survivors (dozens of times)
- The ratio matches the filtering rate: ~90% of behaviors fail fast anyway

For agent orchestration, this means:
- High-frequency decisions (routing, skill selection) need cheap heuristics
- Low-frequency decisions (architecture changes, skill library updates) justify expensive validation
- The architecture must be designed around this cost structure from the beginning

### 4. Independence Requires Diverse Evidence
The VLM provides independent validation because it observes different modalities (video frames vs. state coordinates). In agent systems:
- Validation should use different input sources than execution (logs vs. metrics vs. user feedback)
- Different reasoning methods (rule-based checks, learned models, human-in-loop)
- Different temporal scales (real-time monitoring vs. batch analysis)

### 5. Explicit Handling of Circular Dependencies
The chicken-egg problem (can't verify success function without learning, can't learn without success function) appears throughout autonomous systems:
- Can't validate task decomposition without attempting subtasks
- Can't determine skill prerequisites without trying to learn them
- Can't assess model performance without deployment data

The ASD solution: sample multiple candidates (3 success functions × 3 reward functions × 3 evolution steps) and use the most successful as templates via RAG. This hedges against individual failures while building up validated patterns.

## Boundary Conditions: When This Pattern Doesn't Apply

Fast/slow verification has limitations:

**High-Stakes, Single-Shot Decisions**: If you only get one attempt (nuclear safety, medical diagnosis), you can't afford a fast/cheap initial filter. All verification must be slow/expensive upfront.

**Rapidly Changing Environments**: Slow verification assumes the world stays stable long enough for the VLM to analyze recorded behavior. In highly dynamic environments, validation must happen in real-time or not at all.

**Absence of Ground Truth**: The pattern requires that slow verification is more reliable than fast. When no ground truth exists (creative tasks, preference learning), you need different architectures—perhaps ensemble agreement or human-in-loop.

**Computational Constraints**: The approach assumes slow verification is feasible at all. On resource-constrained edge devices, you might only afford fast verification, requiring more conservative (higher precision, lower recall) initial filters.

## Implementation Guidance for WinDAG Systems

For a DAG-based orchestration system with 180+ skills:

**At Skill Registration Time** (slow verification):
- Independent validator agent reviews skill behavior on test cases
- Different reasoning method than skill implementation (e.g., formal verification for code, VLM for behavior)
- High bar for admission: require multiple validators to agree
- Record validation evidence with skill for future auditing

**At Skill Execution Time** (fast verification):
- Lightweight checks: type validation, precondition testing, timeout guards
- Fail-fast on obvious problems but don't block uncertain cases
- Log fast-check failures for later slow verification
- Use cheap redundancy: run multiple skill options in parallel when stakes are high

**At Composition Time** (medium verification):
- When chaining skills into complex tasks, validate interfaces match
- Check that postconditions of skill N satisfy preconditions of skill N+1
- This is faster than full slow verification but more thorough than fast checks
- Compositional verification: prove properties of the chain from properties of components

**At System Evolution Time** (slowest verification):
- Periodically review skill library for contaminated entries (high false-positive risk)
- Look for skills that frequently fail when used despite passing validation
- Use actual execution success rates as slow-slow verification of slow verification
- Be willing to demote or remove skills that prove unreliable in practice

## The Deeper Lesson: Trust Requires Redundancy

The fast/slow pattern embodies a profound principle: **trustworthy autonomous systems require redundant verification using diverse methods**. A single evaluation, no matter how sophisticated, remains vulnerable to systematic bias. 

The ASD framework shows that even state-of-the-art LLMs, when asked to judge their own outputs, produce unreliable assessments 53.57% of the time (100% - 46.43%). Adding a second, independent evaluation method increases reliability to 76.50%—a 57% reduction in error rate.

For multi-agent systems solving complex problems, this suggests that verification diversity is as important as skill diversity. Just as having multiple ways to solve a problem (skill options) makes systems more robust, having multiple ways to validate solutions makes systems more reliable.

The architecture of intelligence, it seems, requires not just the ability to act but the ability to doubt—and to verify those doubts through independent means.
