# Event-Driven Reactivity: How BDI Agents Respond to a Changing World Without Losing Their Goals

## The Event as the Fundamental Unit of Change

In AgentSpeak(L), the fundamental unit of change is the *event* — not a message, not a command, not a percept, but a formally typed event representing the addition or deletion of a belief or goal. Rao defines four fundamental event types:

- `+b(t)`: A belief was added (something became true in the world)
- `-b(t)`: A belief was deleted (something ceased to be true)
- `+!g(t)`: An achievement goal was adopted (the agent now wants to bring about g(t))
- `-!g(t)`: An achievement goal was abandoned (the agent no longer wants g(t))

And similarly for test goals. This event vocabulary is minimal but complete: it covers every way an agent's relevant world state can change.

Events come in two varieties:
- **External events**: Generated by the environment or other agents, arriving with the "true intention" T as their associated intention. These represent genuine environmental changes.
- **Internal events**: Generated by the agent's own plan execution, arriving with an existing intention. These represent sub-goal adoption during plan execution.

This distinction is architecturally critical. External events may create new intentions (new threads of activity). Internal events extend existing intentions (sub-goals within existing threads). Conflating them destroys the structural coherence of the intention set.

## The Agent Cycle: Observe, Select, Match, Execute

The operational semantics of AgentSpeak(L) defines a tight loop — what practitioners call the *perception-action cycle* or *agent cycle*. Following Rao's interpreter (Figure 1):

1. **Wait for events**: If E is empty, halt. Otherwise, continue.
2. **Select an event**: SE selects an event `<d, i>` from E and removes it from E.
3. **Find applicable plans**: All plans whose triggering event unifies with `d` and whose context is satisfiable from B are the applicable plans (options) O_δ.
4. **Select a plan**: SO selects one plan from O_δ.
5. **Adopt as intention**: If the event is external, create a new intention from the selected plan. If internal, push the plan onto the existing intention i.
6. **Select an intention to execute**: SI selects an intention from I.
7. **Execute the first step of the selected intention**:
   - If it's an achievement goal: generate an internal event, add to E
   - If it's a test goal: find a unifier against B, apply to remaining body
   - If it's an action: add to A (the action queue), remove from body
   - If it's `true`: pop the intention frame, resume parent
8. **Go to step 1**.

This cycle has a crucial property: **every cycle processes at most one event and advances at most one intention one step**. This granularity guarantees interruptibility — between any two steps, new events can arrive and be processed.
